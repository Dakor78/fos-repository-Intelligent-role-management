## Практическая работа, часть 1: «Иерархическая МАС против одиночного ReAct-агента» (облачный LLM API)

> **Формат:** индивидуальная (или в паре) инженерно-исследовательская работа. Студент реализует два решения одной прикладной задачи — **одиночного ReAct-агента** и **иерархическую мультиагентную систему** (планировщик + специалисты + координатор + критик) — и проводит **сравнительное исследование** по качеству, стоимости и времени.
>
> **Вычислительная платформа части 1 — облако.** Весь инференс выполняется через **удалённый LLM API** (OpenAI-совместимый эндпоинт). Развёртывание тех же агентов и локальных моделей на **краевых вычислителях** — предмет [части 2](kim-02-practical-work-edge.md).

### 1. Назначение и место в модуле

Часть 1 опирается на темы 8–11 модуля:

- **Тема 8** (цикл TAO): одиночный ReAct-агент как базовая точка отсчёта (baseline).
- **Тема 9** (принципы и типы взаимодействия МАС): декомпозиция, специализация, коммуникация, координация, верификация, обратная связь; обоснование типа взаимодействия.

**Учебная цель.** Сформировать умение обоснованно выбирать между одиночным агентом и мультиагентной архитектурой, реализовывать иерархическую МАС с разделением ролей и **измеримо** доказывать целесообразность выбора, а не постулировать её.

### 2. Проверяемые результаты обучения

По итогам части 1 обучающийся:

1. **Реализует** одиночного ReAct-агента с явной трассировкой цикла «мысль — действие — наблюдение».
2. **Проектирует и реализует** иерархическую МАС: планировщик со структурированным выводом плана, специалистов с ограниченными наборами инструментов, координатор-агрегатор, агент-критик.
3. **Организует** межагентное взаимодействие по строгому контракту (типизированные сообщения, идентификация отправителя, статусы) в иерархической топологии.
4. **Разграничивает протоколы** «агент–инструмент» (MCP) и «агент–агент» (A2A) и обосновывает их роли.
5. **Обосновывает выбор типа межагентного взаимодействия** (коллаборация, дебаты, конкуренция) для своей задачи.
6. **Проектирует методику измерения** и собирает метрики качества, стоимости (число вызовов модели и токенов) и времени на общем наборе тестовых задач.
7. **Формулирует обоснованные выводы** о границах применимости одиночного агента и МАС, опираясь на полученные данные.

Каждый результат наблюдаем и проверяем по коду, трассам и отчёту.

### 3. Постановка задачи

Студент выбирает **прикладную задачу**, для решения которой требуется **декомпозиция на несколько подзадач** и **несколько групп инструментов** (ориентир — свыше 5–10 инструментов, при котором надёжность одиночного агента заметно падает; см. тему 9). Оба решения (ReAct и МАС) работают над **одной и той же задачей** и используют **один и тот же набор инструментов** — различается только архитектура. Это обеспечивает корректную сравнимость.

Весь инференс — через **удалённый LLM API** (OpenAI-совместимый эндпоинт); параметры доступа студент настраивает самостоятельно.

**Требования к предметной области:**
- задача допускает **проверяемый итог** (можно объективно определить, решена ли она);
- присутствуют подзадачи, естественно распределяемые между специалистами;
- есть сценарии, где одиночному агенту заведомо трудно (много инструментов, смешение контекстов подзадач);
- есть сценарии с **высокой ценой ошибки**, оправдывающие введение критика.

Рекомендуемые предметные области — см. Приложение А; готовые варианты для раздачи — в файле [appendix-01-task-variants.md](appendix-01-task-variants.md) (10 для одиночного агента + 10 мультиагентных систем). Выбранная предметная область и набор инструментов **переиспользуются в части 2**.

### 4. Что необходимо реализовать

#### 4.1. Базовое решение — одиночный ReAct-агент

Один агент с полным набором инструментов, работающий по циклу **TAO** (Thought → Action → Observation) с явной трассировкой каждого шага. Механизм останова и разбора (stop-and-parse) обязателен. Это эталон для сравнения.

#### 4.2. Иерархическая мультиагентная система

Четыре роли в **иерархической** топологии (специалисты **не** взаимодействуют напрямую — только через координатор):

| Роль | Ответственность | Инструменты |
|------|-----------------|-------------|
| **Планировщик** | Декомпозирует задачу в **структурированный план** подзадач с приоритетами и зависимостями (через структурированный вывод). Не исполняет действия. | нет (только рассуждение) |
| **Специалисты** (2–4) | Решают назначенные подзадачи. Каждый имеет **ограниченный** набор инструментов своей специализации. | подмножество инструментов |
| **Координатор** | Назначает подзадачи специалистам согласно плану, собирает результаты, формирует итоговый ответ. Единственная точка агрегации. | оркестрация |
| **Критик** (добавляется на 2-й итерации) | Верифицирует агрегированный результат до финализации; при обнаружении дефекта инициирует доработку (паттерн рефлексии: оценка → при неуспехе повтор с обратной связью), **с обязательным ограничением числа перепроверок** во избежание зацикливания. Модель критика — **не слабее** оцениваемых агентов: слишком слабый критик и пропускает реальные ошибки, и выдумывает несуществующие. | проверочные |

**Порядок построения:** сначала система без критика (планировщик + специалисты + координатор), затем — **добавление критика** и повторное измерение, чтобы показать его вклад.

**Планировщик и координатор** можно реализовать как **две отдельные роли** (как выше) или объединить функции планирования и агрегации в **одном агенте-координаторе** (планирование → делегирование → синтез) — обе схемы допустимы, если декомпозиция выполняется структурированным выводом. Независимые подзадачи плана (без зависимостей в `depends_on`) допустимо выполнять **параллельно** (паттерн handoff с передачей контекста), зависимые — последовательно.

**План планировщика** задаётся структурированным выводом. Минимальный контракт плана (иллюстрация формата, не реализация):

```json
{
  "goal": "исходная цель",
  "subtasks": [
    {"id": "s1", "description": "...", "assignee_role": "specialist_orders",
     "priority": 1, "depends_on": []},
    {"id": "s2", "description": "...", "assignee_role": "specialist_policy",
     "priority": 2, "depends_on": ["s1"]}
  ]
}
```

**Контракт межагентного сообщения** (типизированный, со статусом и идентификацией отправителя):

```json
{"from": "specialist_orders", "to": "coordinator", "subtask_id": "s1",
 "status": "ok | failed | needs_input", "result": { }, "notes": "..."}
```

#### 4.3. Тип взаимодействия и разграничение протоколов (MCP / A2A)

Обязательные проектные решения, оформляемые в отчёте:

- **Обоснование типа межагентного взаимодействия** (тема 9). Явно указать, что выбрана **коллаборация** с передачей подзадачи вместе с контекстом (handoff), и объяснить, почему для данной задачи она предпочтительнее **дебатов** (взаимная критика в несколько раундов) и **конкуренции** (независимые решения с выбором лучшего) по критериям качества, стоимости и времени.
- **Разграничение двух протоколов** (тема 11): взаимодействие **«агент–инструмент»** реализуется как вызов инструментов (в стиле **MCP**), а взаимодействие **«агент–агент»** (планировщик ↔ координатор ↔ специалисты ↔ критик) — как обмен сообщениями по контракту **A2A**. В отчёте показать, какие обмены относятся к каждому протоколу, и почему их не следует смешивать.

*(Факультативно, повышенный уровень: добавление специалиста иной модальности — обработка изображения на основе VLM или голосовой ввод — и его интеграция в ролевую схему.)*

#### 4.4. Требования к трассировке

Обе системы ведут **сквозную трассу** исполнения: для МАС — план, назначения, сообщения между ролями, вызовы инструментов, вердикт критика; для ReAct — полная цепочка TAO. Трасса необходима для отладки и для локализации ошибки в цепочке (тема 10).

### 5. Экспериментальное сравнение (ядро работы)

#### 5.1. Набор тестовых задач

Не менее **8–12** задач одной предметной области, размеченных по **возрастающей сложности** (по образцу темы 8):
- **одношаговые** — решаются одним действием;
- **многошаговые с явной последовательностью** — шаги заданы в запросе;
- **многошаговые с описанием только целевого результата** — агент сам выстраивает план;
- **граничные/негативные** — с высокой ценой ошибки либо требующие воздержаться от действия.

Для каждой задачи фиксируется **ожидаемый итог** (эталон). Набор задач **переиспользуется в части 2** для сравнения облака и края.

#### 5.2. Измеряемые метрики

| Группа | Метрика | Как измеряется |
|--------|---------|----------------|
| **Качество** | доля правильно решённых задач | **объективная (fact-based) проверка**: наличие в ответе конкретных ожидаемых фактов (цены, сборы, идентификаторы, политики) и итогового состояния; без LLM-судьи (он изучается в модуле 4) |
| **Стоимость** | число обращений к модели; суммарные токены (промпт + ответ); денежная оценка | из трассы/учёта вызовов |
| **Время** | время исполнения задачи (включая сетевую задержку API) | замер wall-clock |

Сравниваются **три конфигурации**: (1) одиночный ReAct-агент, (2) иерархическая МАС без критика, (3) иерархическая МАС с критиком. Измерения проводятся на **одном** наборе задач.

#### 5.3. Ожидаемый аналитический результат

Таблица и графики «конфигурация × метрика», а также **выводы о границах применимости**: где МАС оправдывает рост стоимости и задержки (сложные многошаговые задачи, много инструментов, высокая цена ошибки), а где достаточно одиночного агента (простые задачи, мало инструментов). Обязательно — интерпретация **той границы сложности**, за которой одиночный агент начинает деградировать.

### 6. Антипаттерны, которых следует избегать

Часть 1 проверяет и осознанное избегание типовых ошибок проектирования (тема 10):
- избыточное усложнение архитектуры (overengineering) там, где хватает одиночного агента;
- отладка системы целиком вместо поэтапной проверки отдельных агентов;
- отсутствие критика в сценариях с высокой ценой ошибки;
- единая дорогая модель для всех ролей вместо дифференцированного подбора (планировщик/критик — мощнее, типовые специалисты — экономичнее);
- отложенное внедрение трассировки и наблюдаемости.

*(Дифференцированный подбор моделей под роли на облачном API — желательный элемент части 1; в полном объёме, включая привязку моделей к устройствам, он выполняется в части 2.)*

### 9. Правила использования внешних ресурсов и генеративного ИИ

- Большие языковые модели — **предмет и инструмент** работы: их штатное использование в агентах обязательно; параметры доступа к облачному API студент настраивает самостоятельно.
- Разрешено использовать библиотечные каркасы агентов (например, класса LangGraph) и открытые инструменты; **архитектурные решения, распределение ролей и методика сравнения должны быть авторскими** и объяснимыми на защите.
- Внешние данные и инструменты должны иметь устойчивые ссылки и корректные условия использования.
- Запрещается выдавать чужой код или результат за свой; заимствования оформляются ссылками.

### 10. Методические указания (рекомендуемые этапы)

1. Выбрать предметную область и зафиксировать общий набор инструментов (Приложение А).
2. Реализовать одиночный ReAct-агент, убедиться в корректной трассе TAO.
3. Сформировать набор тестовых задач с эталонными итогами и разметкой по сложности.
4. Реализовать иерархическую МАС без критика; задать контракт межагентных сообщений и разграничить обмены MCP (агент–инструмент) и A2A (агент–агент); наладить сквозную трассировку.
5. Обосновать выбранный тип взаимодействия против дебатов и конкуренции.
6. Добавить критика; повторить измерения.
7. Провести замеры всех метрик на трёх конфигурациях; построить таблицу и графики.
8. Сформулировать выводы о границах применимости; подготовить защиту.
9. Сохранить систему, набор инструментов и набор задач для **части 2**.

### Приложение А. Рекомендуемые предметные области

- **Ассистент службы поддержки** с комплексным запросом (например, частичный возврат + замена товара + уточнение сроков доставки замены): специалисты по заказам, по политикам (RAG), по расчёту возврата, по логистике.
- **Аналитический ассистент по набору данных**: планировщик разбивает вопрос на подзапросы; специалисты выполняют выборку, агрегацию, проверку; координатор собирает отчёт.
- **Ассистент подготовки обзора/справки** по корпусу документов: специалисты по поиску, извлечению фактов, сверке источников; критик проверяет обоснованность утверждений.
- **Киберфизический сценарий** (удобен для перехода к части 2): контроль доступа/мониторинг помещения/управление инженерной системой — специалисты-исполнители, супервизор, администратор, диагностика оборудования.

Требование ко всем вариантам: наличие подзадач для распределения, свыше 5–10 инструментов суммарно и проверяемый итог.
