# Несколько LLM вместе: каскад, ансамбль и подбор размеров моделей

Материал дополняет модуль 3. Он показывает не только классические мультиагентные архитектуры, но и инженерные схемы совместного использования нескольких моделей: дешёвая модель как router, средняя модель как исполнитель, большая модель как арбитр или критик.

## 3.1. Каскад: малая модель → эскалация на большую

### Цель

Каскад используется, когда главная цель — **снизить стоимость** и не отправлять все запросы на дорогую большую модель.

### Схема

```text
запрос
  ↓
router / малая модель / эвристика
  ↓
решение: обработать дешёвой моделью или эскалировать
  ↓                         ↓
малая модель              большая модель
  ↓                         ↓
ответ + confidence          ответ
  ↓                         ↓
логирование качества, цены, латентности
```

### Компоненты

| Компонент | Назначение |
|---|---|
| Router | определяет сложность запроса, риск ошибки, наличие запрещённых данных, необходимость инструментов |
| Cheap executor | обрабатывает простые и типовые запросы |
| Strong executor | обрабатывает сложные, рискованные, юридически значимые или низкоуверенные запросы |
| Escalation policy | формализует критерии перехода на большую модель |
| Metrics logger | считает качество, стоимость, latency, долю эскалаций |

### Критерии эскалации

Запрос эскалируется на большую модель, если выполняется хотя бы одно условие:

- router относит задачу к высокой сложности;
- дешёвая модель вернула низкую уверенность или противоречивый ответ;
- запрос требует нескольких инструментов или длинного планирования;
- ответ касается безопасности, финансовых последствий, юридических ограничений;
- в ответе дешёвой модели найдено нарушение формата;
- проверочный critic не принимает ответ;
- пользователь повторно задаёт тот же вопрос после неудачного ответа.

### Стоимость в вызовах

| Сценарий | Вызовы |
|---|---:|
| Простая задача, обработана малой моделью | 1 вызов router/малой модели или 1 совмещённый вызов |
| Сложная задача, эскалация | 1 вызов router + 1 вызов большой модели |
| Каскад с critic | 1 router + 1 executor + 1 critic |
| Каскад с повторной генерацией | 1 router + 1 executor + 1 critic + 1 reviser/strong |

### Латентность

Каскад снижает среднюю стоимость, но может увеличить latency на эскалированных запросах: сначала выполняется router, затем большая модель. Поэтому router должен быть очень лёгким: эвристика, классификатор, модель 0.5–3B или дешёвый API.

### Спекулятивное декодирование

Спекулятивное декодирование похоже по идее «малая помогает большой», но решает другую задачу.

| Подход | Цель | Как работает |
|---|---|---|
| Каскад | снизить цену за счёт обработки части запросов малой моделью | малая модель решает, нужна ли большая |
| Speculative decoding | ускорить генерацию большой модели | малая draft-модель предлагает токены, большая их проверяет |

В курсе это можно рассматривать как соседний приём: router экономит вызовы большой модели, speculative decoding экономит время генерации при вызове большой модели.

### Типичный провал каскада

Главный риск — **ошибка router**: сложная задача ошибочно считается простой, дешёвая модель даёт уверенный, но неверный ответ. Поэтому каскад обязательно тестируется не только по цене, но и по качеству на сложных и пограничных задачах.

## 3.2. Ансамбль: параллельные модели + агрегация

### Цель

Ансамбль используется, когда главная цель — **повысить качество / устойчивость**, а не снизить стоимость.

Общая схема:

```text
запрос
  ↓
несколько независимых генераций / агентов / моделей
  ↓
агрегация: голосование, judge, critic, reviser, weighted merge
  ↓
итоговый ответ + обоснование + лог стоимости
```

### Обзор техник

| Техника | Схема | Когда применима | Стоимость в вызовах | Латентность | Типичный провал |
|---|---|---|---:|---|---|
| Voting / self-consistency | N ответов одной модели → голосование / выбор наиболее частого вывода | задачи с проверяемым кратким ответом, рассуждение, математика, классификация | N | при параллельном запуске близка к одному медленному вызову; при последовательном — N× | большинство может быть одинаково неверным |
| LLM-as-a-judge | несколько ответов → judge выбирает лучший по рубрике | оценка открытых ответов, сравнение вариантов, контроль качества | N + 1 | добавляется вызов judge | judge наследует смещения и может предпочитать более уверенный стиль |
| Mixture-of-Agents | несколько агентов генерируют черновики → следующий слой агентов улучшает/синтезирует | сложные аналитические задачи, где полезны разные перспективы | N × L + aggregator | высокая, особенно при нескольких слоях | агенты начинают повторять друг друга, растёт конформизм |
| Multi-agent debate | агенты спорят, критикуют аргументы, затем арбитр выбирает | спорные вопросы, проверка гипотез, анализ альтернатив | 2N–kN + arbiter | высокая; зависит от числа раундов | дебаты уходят в риторику, а не в проверку фактов |
| Generator–critic–reviser | generator создаёт ответ → critic находит ошибки → reviser исправляет | генерация инструкций, планов, кода, отчётов | 3 | средняя; обычно последовательная | critic не замечает системную ошибку или reviser вносит новые ошибки |
| Взвешенная агрегация | ответы разных моделей объединяются с весами по доверенности/метрикам | разные модели сильны в разных классах задач | N + aggregator | средняя/высокая | веса невалидны вне тестового набора; слабая модель портит итог |

### Общий вывод по ансамблям

Ансамбль не гарантирует истину. Он увеличивает разнообразие сигналов, но также может усилить **конформизм**: модели обучены на похожих данных, склонны к похожим формулировкам и могут согласованно ошибаться. Поэтому ансамбль должен сопровождаться:

- независимыми критериями проверки;
- тестовыми задачами с эталоном;
- логированием расхождений;
- анализом случаев, где все модели ошиблись одинаково;
- контролем стоимости и latency.

## 3.3. Размеры моделей и требования к железу

Ориентиры для проектирования ролевой LLM-системы:

| Роль | Типичный размер | Где запускать | Зачем |
|---|---:|---|---|
| Router / classifier | 0.5–3B или эвристика | CPU, малый GPU, дешёвый API | быстро решить: дешёвый путь или эскалация |
| Cheap executor | 3–7B | локальный GPU 8–16 ГБ, edge, дешёвый API | типовые ответы, извлечение, маршрутизация |
| Main executor | 7–14B | GPU 16–24 ГБ при INT4/INT8, облачный API | основной ответ на большинство задач |
| Critic / verifier | 7–14B или 30B+ | GPU 24–48+ ГБ, API | проверка формата, фактов, безопасности |
| Arbiter / judge | 30–70B или сильная закрытая модель | multi-GPU / облачный API | финальное сравнение сложных ответов |
| Long-context analyst | 7–70B с длинным контекстом | зависит от KV-cache | анализ длинных документов и трасс |

### Что это значит для архитектуры

- Router должен быть дешёвым и быстрым. Если router дорогой, каскад теряет смысл.
- Исполнитель 7–14B часто является разумным компромиссом для локального/гибридного контура.
- Арбитр 30–70B нужен только там, где выигрыш качества оправдывает стоимость.
- На 24 ГБ VRAM практичнее запускать 7–14B в INT4/INT8, чем пытаться разместить 70B.
- Для длинного контекста нужно считать не только веса, но и KV-cache.

## 3.4. Практика: демо каскада

Демонстрация находится в ноутбуке:

```text
attachments/examples/07_cascade_routing_demo.ipynb
```

Идея практики:

1. Сформировать набор задач разной сложности.
2. Запустить стратегию `all_strong`: все задачи решает большая модель.
3. Запустить стратегию `cascade`: router отправляет простые задачи дешёвой модели, сложные — большой.
4. Сравнить:
   - среднюю стоимость;
   - долю эскалаций;
   - качество;
   - latency;
   - типы ошибок router.
5. Сделать вывод: при какой доле простых задач каскад экономит деньги без существенной потери качества.

Минимальная таблица отчёта:

| Стратегия | Среднее качество | Стоимость | Latency | Доля эскалаций | Типичный провал |
|---|---:|---:|---:|---:|---|
| Всё на большой модели | | | | 100% | дорого, но устойчиво |
| Каскад | | | | | router может ошибиться |
| Каскад + critic | | | | | дороже, но безопаснее |

## 3.5. Что должно войти в вывод студента

Студент должен показать не только выигрыш в цене или качестве, но и границы применимости:

```text
Каскад выгоден, если большая доля запросов простая и router редко ошибается.
Ансамбль полезен, если цена и latency допустимы, а качество действительно растёт.
Несколько моделей вместе не отменяют необходимость оценки: ошибки могут быть согласованными.
```
