# Модуль IV — примеры (Python-ноутбуки)

Пошаговая сборка системы оценки качества (eval) для агента — по одному компоненту на тетрадь. Сквозной домен — поддержка интернет-магазина со **средой, меняющей состояние** (объект оценивания). Ноутбуки самодостаточны и запускаются сверху вниз; корзина задач сохраняется на шаге 2 и переиспользуется дальше.

## Ноутбуки

| Файл | Тема | Содержание |
|------|------|-----------|
| `01_agent_and_environment.ipynb` | — | Объект оценивания: среда с изменяемым состоянием, инструменты, политики, агент с записью траектории; две версии агента |
| `02_task_suite.ipynb` | 13 | Корзина задач: полная структура, ручная сборка + генерация БЯМ по матрице, валидация, баланс, версионирование |
| `03_harness_and_graders.ipynb` | 14 | Eval harness (изоляция, параллельность, трассировка); программные грейдеры (state/tools/safety); Iron User; pass@k / pass^k |
| `04_llm_judge_kappa.ipynb` | 15 | LLM-судья для смысловых критериев; человеческий эталон; калибровка по каппе Коэна; итерации промпта; агрегация «блокирующие + взвешенные» |
| `05_regression_and_ops.ipynb` | 16 | Регрессия двух версий; экономика (лёгкая vs мощная модель); пороги качества (go/no-go); анализ хвостов |
| `06_public_benchmark_tau2.ipynb` | 15 | *(опционально)* Публичный бенчмарк **τ²-bench**: прогон агента на готовом наборе задач через тот же эндпоинт; pass@k / pass^k |

Вспомогательные модули:
- `eval_env.py` — среда, инструменты, политики, ReAct-агент с траекторией (общий для всех тетрадей).
- `eval_harness.py` — параллельный прогон корзины и программные грейдеры (примеры 4–5).
- `data/eval_suite.json` — корзина задач (создаётся примером 2).

## Установка

Python 3.12. Зафиксированные версии — в `requirements.txt`:

```bash
pip install -r requirements.txt
```

## Доступ к модели (через переменные окружения)

Ключ и адрес **не вшиты в код** — читаются из окружения при запуске:

```bash
export LLM_API_KEY=<ваш-ключ>
export LLM_BASE_URL=https://<эндпоинт>/v1
export LLM_MODEL=<модель>                 # напр. qwen/qwen3.7-max
export LLM_MODEL_CHEAP=<лёгкая-модель>     # напр. openai/gpt-4.1-nano (для примера 05)
```

Запуск одной тетради (сначала выполните пример 2, чтобы создать корзину):

```bash
jupyter nbconvert --to notebook --execute --inplace 02_task_suite.ipynb
```

Подойдёт любой OpenAI-совместимый провайдер; можно экспериментировать с более лёгкими моделями.

## Пример 06: публичный бенчмарк τ²-bench

Пример 6 — **опциональный/повышенный**: он подключает внешний публичный бенчмарк **τ²-bench** и прогоняет на нём агента через тот же OpenAI-совместимый эндпоинт. Это иллюстрация того, что корзину задач не обязательно писать с нуля — существуют готовые эталонные наборы (capability eval).

### Что это и зачем создан

**τ²-bench** (репозиторий [sierra-research/tau2-bench](https://github.com/sierra-research/tau2-bench), лицензия MIT) — бенчмарк для оценки **диалоговых tool-агентов** службы поддержки. Развитие более раннего τ-bench. Задача бенчмарка — измерять не разовую удачу, а **надёжность** агента в реалистичных многоходовых сценариях, где агент:

- ведёт диалог с **симулируемым пользователем**,
- вызывает инструменты, меняющие **состояние базы данных**,
- обязан соблюдать **политики** предметной области,
- а успех проверяется по **итоговому состоянию среды**, а не по тексту ответа.

Ключевая идея (и в τ²-bench особый акцент) — **dual-control**: в некоторых доменах действия может совершать не только агент, но и пользователь, что делает сценарии ближе к реальности. Отсюда упор на метрику **pass^k** (успех во всех попытках) — единичный успех не означает стабильного поведения.

### Что внутри репозитория

- **Домены** (`data/tau2/domains/…`): `airline`, `retail`, `telecom`, служебные `mock` и `banking_knowledge`. Каждый домен содержит: документ политик (`policy.md`), набор задач (`tasks.json` с ожидаемым итоговым состоянием и эталонным решением), базу данных домена (`db.json`) и данные пользователя.
- **Симулятор пользователя** (`user_simulator`, роли `user_simulator` / `dummy_user`) — генерирует реплики пользователя по сценарию, не выдумывая данных вне контекста.
- **Агенты** (`llm_agent`, `llm_agent_gt` — по эталонному решению, `llm_agent_solo`).
- **Оркестратор и runner** — поднимают изолированную среду, ведут многоходовой диалог, ограничивают число шагов, поддерживают повторы и параллельность.
- **Оценщик и метрики**: проверка итогового состояния БД (**DB match**), проверка действий (read/write actions), reward (компоненты COMMUNICATE + DB), **LLM-judge review**, а также эксплуатационные метрики — стоимость на диалог, `pass@k` / `pass^k`.
- Доступ к моделям — через **litellm**, поэтому бенчмарк работает с любым OpenAI-совместимым эндпоинтом.

### Как использовать

1. **Установка** (в PyPI одноимённого пакета нет — ставится из GitHub; тянет `litellm`):
   ```bash
   pip install "git+https://github.com/sierra-research/tau2-bench.git"
   ```
2. **Данные.** Полный набор данных большой (~700 МБ, в основном примеры прогонов). Ноутбук 06 подтягивает **только домен retail** через sparse-checkout (~33 МБ) и указывает путь переменной окружения `TAU2_DATA_DIR`.
3. **Модель.** Бенчмарк вызывает модель через litellm: в ноутбуке переменные `LLM_API_KEY` / `LLM_BASE_URL` пробрасываются в `OPENAI_API_KEY` / `OPENAI_API_BASE`, а имя модели берётся в формате litellm (`openai/<модель>`, напр. `openai/gpt-4.1-mini`).
4. **Запуск.** Программный API: `TextRunConfig(domain=..., llm_agent=..., llm_user=..., num_tasks=..., num_trials=k)` → `run_domain(config)`; из результата берутся `reward` по каждой попытке, из которых считаются `pass@k` / `pass^k`. Есть и CLI (`tau2 run --domain retail --num-tasks ...`).

В ноутбуке всё это уже настроено: он проверяет установку пакета, подтягивает данные, направляет litellm в ваш эндпоинт, прогоняет 2 задачи по 3 попытки и показывает `pass@k` / `pass^k`. Бенчмарк намеренно трудный, поэтому низкий `pass^k` — ожидаемый и показательный результат.
