# Модуль I — примеры (Python-ноутбуки)

Учебные примеры к темам 2–3. Сквозной домен — поддержка интернет-магазина «ШопБот». Каждый ноутбук самодостаточен, снабжён подробными пояснениями и запускается сверху вниз.

Логика последовательности: сначала БЯМ разбирается как **вычислительное ядро** с измеримыми характеристиками и границами (примеры 01–03), затем вокруг неё собирается **агент** — сначала вручную, потом на каркасе (примеры 04–05).

## Ноутбуки

| Файл | Тема | Содержание |
|------|------|-----------|
| `01_tokenization_and_context.ipynb` | 2 | Символы, слова и токены; сравнение двух кодировок; цена русского языка; специальные токены и их опасность в пользовательском вводе; расчёт вместимости контекстного окна и стоимости запроса |
| `02_decoding_and_reproducibility.ipynb` | 2 | Распределение вероятностей (logprobs); температура и разброс ответов; почему `temperature=0` не даёт строгих гарантий; `top_p`; подбор параметров под тип роли |
| `03_llm_limits_and_failures.ipynb` | 2 | Типовые отказы: арифметика, срез знаний, галлюцинации, чувствительность к формату; компенсация инструментом; сравнение «алгоритм против БЯМ» на одной операции |
| `04_basic_agent_loop.ipynb` | 3 | Базовый агент **вручную**: ролевая системная инструкция из 4 блоков, уровни `system`/`user`/`assistant`/`tool`, цикл исполнения, лимит итераций, обработка ошибок, полная трасса; три сценария (успех, отказ, деградация) |
| `05_agent_execution_graph_langgraph.ipynb` | 3 | Тот же агент как **граф исполнения** на LangChain/LangGraph: `@tool`, `StateGraph`, `ToolNode`, условное ребро; пошаговое наблюдение цикла; построчное сопоставление с ручной реализацией |

Пример 04 при запуске создаёт файл `trace_example.json` — образец сохранённой трассы (артефакт, сдаваемый в [КИМ-1.3](../../kim-lab-03.md)).

## Установка

Python 3.12. Зафиксированные версии — в `requirements.txt`:

```bash
pip install -r requirements.txt
```

Отдельные датасеты не нужны — учебный домен реализован в памяти внутри ноутбуков.

## Доступ к модели (через переменные окружения)

Ключ и адрес **не вшиты в код** — читаются из окружения при запуске:

```bash
export LLM_API_KEY=<ваш-ключ>
export LLM_BASE_URL=https://<ваш-openai-совместимый-эндпоинт>/v1
export LLM_MODEL=<имя-модели>            # напр. openai/gpt-4.1-mini
```

Запуск одного ноутбука:

```bash
jupyter nbconvert --to notebook --execute --inplace 01_tokenization_and_context.ipynb
```

Или открыть в Jupyter/VS Code и выполнить ячейки по порядку (задав переменные окружения заранее).

**Пример 01 работает без обращения к модели** — токенизация выполняется локально, ключ для него не нужен.

### Требования к модели

Примеры не привязаны к конкретной модели; подойдёт любая, удовлетворяющая условиям:

- OpenAI-совместимый эндпоинт `chat/completions`;
- **поддержка вызова инструментов** (tool calling) — обязательна для примеров 04–05;
- поддержка `logprobs` — желательна для примера 02 (иначе раздел 1 этого ноутбука пропускается);
- можно экспериментировать с более лёгкими и дешёвыми моделями.
